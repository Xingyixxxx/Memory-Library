# 记忆博物馆（Memory Museum）产品落地方案

## 1. 项目定位

这不是一个大众化社交产品，而是一个**偏个人使用的、可分享给亲友或伴侣观看的沉浸式数字回忆馆**。它的核心不是“效率工具”，而是“情绪化记录 + 美学化浏览 + 可反复回看”。

它应该同时具备三层价值：

1. **记录层**：把照片、语音、文字、音乐组织成一条条有温度的记忆。
2. **浏览层**：不是普通相册，而是一个带有 3D 空间感、粒子氛围和章节结构的“数字展厅”。
3. **回放层**：可以安静地看，也可以让系统用第一人称语音读出来，像一本会自己讲述的回忆录。

一句话概括：
**这是一个以照片为主、文字为辅、语音和音乐增强氛围的 3D 个人记忆展馆。**

## 3. 功能模块拆解

## 模块一：记忆内容管理

这是整个系统的内容底座。

### 你要管理的对象不是“文件”，而是 4 层内容结构

**主题馆**
例如：日常、旅行、恋爱、灵感、某个人、某个阶段。

**记忆本（Book）**
一本本子代表一个相对独立的叙事单元。
例如：《2025 年夏天》《和她的故事》《东京旅行手记》。

**篇章 / 事件（Entry）**
一本本子里有多个事件。
例如：某天的日记、某次出行、某次争吵、某次晚餐、某个灵感时刻。

**媒体资产（Asset）**
照片、音频、背景音乐、封面图、转写文本、生成朗读音频等。

### 推荐字段设计

每个 Entry 建议至少有：

* `id`
* `book_id`
* `title`
* `subtitle`
* `event_date`
* `location`
* `people`
* `tags`
* `cover_photo`
* `photos[]`
* `voice_original`
* `voice_transcript`
* `narration_text`
* `narration_audio`
* `bgm_audio`
* `mood`
* `visibility`
* `created_at`
* `updated_at`

这里最关键的是：
**照片、音频、文本不要混成一坨，而要分成“媒体文件 + 元数据”两层。**
后期你想改 UI、改排序、改播放逻辑，就不会很痛苦。

## 模块二：3D 交互展厅

这是你最在意、也是最容易被开发误解的部分，所以我把它写清楚。

### 这个模块的目标不是“做一个复杂游戏”

而是做一个**低门槛、可控、稳定的 3D 浏览器内交互空间**。

### 推荐交互形式

#### 方案 1：漂浮书架 / 记忆星系

首页空间里漂浮着若干本“书”或“记忆球体”。
鼠标 hover 时：

* 周围粒子变亮
* 书本或卡片缓慢靠近
* 显示标题、日期、标签
* 背景音乐淡入一层环境音

点击后：

* 摄像机平滑靠近
* 场景做一次轻微转场
* 进入该本子的阅读页

这是最稳的 MVP 方案。

#### 方案 2：照片粒子显影

照片默认并不完整显示，而是由粒子聚合形成。
当鼠标移动到某个记忆节点附近时：

* 粒子向照片轮廓聚拢
* 照片逐渐清晰
* 对应文字淡淡浮现
* 若有语音，可触发“播放回忆”按钮

这个方案很符合你的“粒子显现”想法，但技术上比普通卡片 hover 更复杂，通常要结合 instancing / shader / 粒子系统来做性能优化。Three.js 本身提供底层渲染能力，React Three Fiber 适合把这些对象组织进 React 组件体系。([Poimandres 文档][1])

#### 方案 3：按时间轴排列的旋转记忆长廊

整个空间不是散点，而是一个沿时间推进的长廊。
用户旋转、拖拽、缩放场景时，可以看到不同年份、月份或阶段的章节。

这个适合 V2，因为它需要更明确的叙事结构和镜头设计。

## 模块三：语音转文字（STT）

这个模块很直接，你原本的判断也是对的：
**先全部走 API，不自己部署。**

使用Deepgram的API进行语音转文字。

### 这个模块在你的产品里应该承担 3 件事

1. **把日常语音输入变成文本**
2. **给照片补充上下文**
3. **为后续“自动整理回忆”提供原始语料**

### 推荐流程

用户上传或录制一段语音后：

1. 前端上传音频到服务端
2. 服务端调用 STT API
3. 返回转写文本
4. 用户可手动修改
5. 保存“原始转写”和“修订版文本”两份

这里一定要保存两份，原因很简单：
原始转写用于留痕，修订版用于展示。你以后如果想做“保留原声感”与“文学化叙述”两种模式，就不需要返工。

---

## 模块四：文字转语音（TTS）

这个模块不是刚需，但它是产品“有灵魂”的关键增强项。

使用Gemini的API调用TTS。

### 生成“第一人称回忆旁白”

朗读原文，先把原始文字转成更柔和、连贯、适合口述的叙述版本，再合成音频。

例如：

* 原始文本偏日记
* 展示文本偏书面
* 朗读文本偏旁白

这会让体验更像一个“个人纪录片”。

这个旁白的原文可以直接就写出来，不需要再当场生成。

---

## 模块五：背景音乐与氛围系统

这个模块不复杂，但非常影响质感。

### 建议支持 3 种模式

1. **全局环境音乐**
   进入展馆时统一播放轻氛围音。

2. **本子级音乐**
   每一本记忆本可以绑定一首背景音乐。

3. **事件级音乐**
   某个特定事件可以绑定独立音乐。

### 交互原则

* 音乐默认轻，不喧宾夺主
* 切换页面时使用淡入淡出
* 朗读开始时，背景音乐自动降音量
* 用户必须能一键静音
* 是否可以AI自动配乐，轻音乐类别

---

## 模块六：主题与分类系统

你前面说得很清楚：这个产品像“很多本子”。

所以它的分类系统不应该像企业后台，而应该像“个人档案馆”。

### 推荐分类维度

**一级：主题本**

* 日常生活
* 旅行记录
* 恋爱回忆
* 灵感笔记
* 某个人
* 某个城市
* 某个阶段

**二级：事件**

* 日期
* 地点
* 人物
* 情绪
* 天气
* 关键词

### 建议你一定保留的能力

* 通过日期筛选
* 通过人物筛选
* 通过标签筛选
* 通过关键词搜索
* 通过“情绪”筛选

因为你的产品不是简单相册，它以后很可能会变成一个“可检索的个人人生资料库”。

## 4. 推荐技术实现方式

这里我直接给你一条**最适合个人项目起步**的技术路线。

## 前端

### 推荐组合

**Next.js + React + React Three Fiber + Drei + Tailwind CSS**

原因是：

* Next.js 当前官方主推 App Router，适合把页面、上传、服务端逻辑放在同一个项目里组织。([Next.js][5])
* React Three Fiber 是 three.js 的 React renderer，天然适合做“3D 场景 + 普通 React 页面”混合型项目。([Poimandres 文档][1])
* Drei 提供很多常用封装，能减少你们自己重复造轮子。([Poimandres 文档][6])

### 这条路线特别适合你的原因

因为你的产品不是纯网站，也不是纯 3D demo。
它是：

* 一部分是 3D 空间
* 一部分是普通内容页
* 一部分是上传 / 播放 / 管理
* 还要接 API

所以不用把前端拆成很多套，直接用 Next.js 统一比较省心。

---

## 后端

### 推荐组合

**Next.js Route Handlers / Server Actions + 独立媒体处理逻辑**

核心职责：

* 接收照片和音频上传
* 调用 STT / TTS API
* 存数据库
* 生成页面数据
* 做权限判断

如果一开始规模不大，其实不用先拆独立后端服务。
先在 Next.js 项目内部做服务端逻辑，够用了。
后面如果功能变复杂，再把媒体处理拆出来。

---

## 数据库

### 推荐组合

**Supabase Postgres**

Supabase 官方目前提供完整 Postgres 数据库，并且把 Auth、Storage、Realtime、Functions 放在同一平台内；官方也明确说明每个项目是完整的 Postgres 数据库。([Supabase][7])

### 适合你的原因

你这个项目不是高并发交易系统，核心是：

* 内容结构清楚
* 图片和音频可关联
* 权限清楚
* 后期可扩展

Postgres 非常适合这种“结构化内容 + 关系明确”的场景。

---

## 文件存储

这里我建议二选一。

### 方案 A：Supabase Storage

Supabase Storage 官方定位就是文件存储，并且和其 Auth / Postgres 权限体系联动。([Supabase][8])

适合你如果想：

* 全部放一个平台
* 权限控制简单
* 开发速度快

### 方案 B：Cloudflare R2

Cloudflare 官方说明 R2 是 S3-compatible object storage，并强调 zero egress fees；其文档也说明可以用 S3 兼容 API 和现成工具链访问。([Cloudflare][9])

适合你如果想：

* 以后媒体文件变多
* 控制图片 / 音频外链成本
* 保留更灵活的云存储迁移能力

### 我的建议

第一版为了省事，**优先 Supabase 全家桶**。
如果后面照片、音频、朗读文件明显增多，再迁到 R2。

---

## 身份与权限

### 建议做成 3 级权限

1. **仅自己可见**
2. **有链接 + 密码可见**
3. **邀请制可见**

你这个产品有大量私人内容，所以权限不是附属功能，而是核心功能。

### 技术建议

如果你用 Supabase，Auth 也能一起接进来。其官方文档说明 Auth 底层就是项目的 Postgres 体系，并可与业务表关联。([Supabase][10])

---

## 部署方式

### 推荐部署路径

#### 第一阶段：Vercel 部署前端与服务端逻辑

Vercel 官方文档明确支持 Next.js 部署，并且每次部署都会生成可预览 URL；对于迭代中的个人项目，这一点很有用。([Vercel][11])

#### 第二阶段：你买域名后绑定正式站点

这个很适合你后面把它给亲友或伴侣看。

#### 第三阶段：若你坚持自有服务器

Next.js 官方文档明确写到可以部署为 Node.js server、Docker container、static export 或其他 adapter；但 static export 功能有限。你这个项目有上传、权限、API、动态内容，所以更适合 **Node.js server 或 Docker**，不建议纯静态导出。([Next.js][12])

## 测试工具

### 推荐

**Playwright**

Playwright 官方文档强调其支持 Chromium、WebKit、Firefox，适合做现代 Web 应用的端到端测试。你的项目里最重要的是交互路径是否真的顺：进入首页、hover、点击、打开本子、播放音频、上传内容，这些都适合 E2E 测试。([Playwright][13])

## 5. 你这个项目应该使用哪些“产品”

如果你要和别人讲“我要买什么 / 用什么”，可以直接按这份说：

### 必选产品

* **Next.js**：整个 Web 应用框架
* **React Three Fiber + Three.js**：3D 展厅与粒子交互
* **OpenAI Audio API**：语音转文字、文字转语音
* **Supabase**：数据库、登录、文件存储
* **Vercel**：测试环境与正式部署

### 可选产品

* **Cloudflare R2**：后期媒体文件量变大时的对象存储
* **独立 TTS 服务**：如果后面发现中文朗读效果需要更强，可替换
* **独立 CMS / 后台编辑器**：如果以后你不想直接在数据库里管理内容
